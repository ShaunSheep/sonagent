# AI 开发辅助规则

> 本文件定义 AI 编程助手在项目开发中的行为规范和代码生成标准。

---

## 1. 沟通规范

### 1.1 需求确认流程
```
用户提出需求 → AI 分析需求 → 更新 PRD → 用户确认 → 执行代码
```

**原则**：先改文档，用户确认后再写代码。

### 1.2 执行开发流程前的检查（强制）

当用户说"执行开发流程"时，**必须**先执行以下步骤：

```
1. 整理本次所有需求和Bug
2. 以表格形式展示出来（编号、类别、内容）
3. 询问用户确认是否有遗漏
4. 得到用户确认后，再次提示"确认无遗漏，是否按开发流程执行？"
5. 用户再次确认后才开始代码开发
```

**示例输出**：
```
## 本次需求/Bug清单

### Bug修复
| # | 问题 | 原因 | 解决方案 |
|---|------|------|----------|
| 1 | xxx功能失效 | xxx | xxx |
| 2 | xxx显示异常 | xxx | xxx |

### 新增需求
| # | 需求 | 说明 |
|---|------|------|
| 3 | xxx功能 | xxx |

---
共3项（2 Bug + 1需求）

请确认是否有遗漏？确认后我将进入开发流程。
```

### 1.3 诚实原则
- 如实报告工作进度，不夸大
- 遇到不确定的问题，明确告知用户
- 主动说明潜在风险和影响

### 1.4 信息反馈
- 完成后说明做了哪些修改
- 如有遗漏，主动承认并补上
- 不报喜不报忧

## 2. 代码规范

### 2.1 文件操作
- **读取优先**：修改文件前先读取文件内容
- **避免覆盖**：使用 `replace_in_file` 工具进行精确替换
- **创建谨慎**：非必要不创建新文件

### 2.2 代码格式
- 使用 ES6+ 语法
- 使用 2 空格缩进
- 使用 `const` / `let` 替代 `var`
- 使用模板字符串拼接字符串

### 2.3 命名规范
| 类型 | 规范 | 示例 |
|------|------|------|
| 变量/函数 | 小驼峰 | `getUserInfo` |
| 常量 | 全大写下划线 | `MAX_COUNT` |
| 类名 | 大驼峰 | `UserService` |
| 文件名 | 小写中划线 | `user-service.js` |

---

## 3. 文档规范

### 3.1 必须维护的文档
| 文档 | 用途 | 更新时机 |
|------|------|----------|
| PRD.md | 需求文档 | 需求变更时 |
| 开发记录.md | 任务清单和方案 | 确认需求后 |
| 沟通记录.md | 沟通纪要 | 每次讨论后 |
| 变更日志.md | 代码变更记录 | 开发完成后 |
| ISSUES.md | 技术问题积累 | 问题解决后 |

### 3.2 版本号规则
- 格式：`vX.Y`
- Y=0 为大版本，Y>0 为小版本
- 沟通轮次：`YYYY-MM-DD-序号`

### 3.3 文档头部信息
```markdown
> **当前版本**: v1.0 | **沟通轮次**: 第1次 | **更新日期**: 2026-04-26
> 
> **版本变更**: v0.9 → v1.0 完成基础功能开发
```



## 5. 安全规范



### 5.2 执行危险命令
- 删除文件前确认路径
- 执行系统命令前说明操作
- 危险操作需要用户批准

---

## 6. AI 自我检查清单

完成工作后，检查以下事项：

- [ ] 是否读取了所有相关文件？
- [ ] 修改是否完整覆盖所有涉及的文件？
- [ ] 是否主动说明了修改的影响？
- [ ] 是否存在遗漏或潜在问题？
- [ ] 文档是否同步更新？
- [ ] 代码是否符合项目规范？

---

## 7. 常见错误避免

| 错误类型 | 错误示例 | 正确做法 |
|----------|----------|----------|
| 文件未读取 | 直接修改未知文件 | 先 read_file |
| 上下文丢失 | 长时间未读文件后编辑 | 编辑前重新读取 |
| 过度修改 | 大量重写代码 | 使用 replace_in_file |
| 遗漏检查 | 修改后不验证 | 使用 read_lints |
